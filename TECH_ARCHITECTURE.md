# VideoToBookStory 技术架构设计（v1）

## 1. 产品目标与约束

### 1.1 目标
将本地视频/电影或 YouTube 链接转换为“儿童友好”的图画书：
- 自动抽取关键画面并按剧情编排页面。
- 生成低龄可读文字（如 4-6 岁）。
- 保留关键情节与转场，避免只“抽美图”导致断裂叙事。
- 支持人工校对、编辑、导出（PDF/图片包）。

### 1.2 非目标（v1）
- 不做复杂多用户协同编辑。
- 不做完整长视频实时流式处理（先做离线任务）。
- 不追求电影级镜头语义理解（先做到“可用且可控”）。

---

## 2. 分层架构总览

建议采用“前后端 + 异步任务 + 对象存储 + 向量/关系数据库”的分层：

1. **Web 前端（React/Next.js）**
   - 上传视频、填写 YouTube 链接。
   - 可视化时间轴与场景列表。
   - 页面级图片/文字编辑。
   - 预览与导出下载。

2. **API 网关（FastAPI/Node）**
   - 鉴权、项目管理、任务提交。
   - 读取/写入元数据（项目、页面、版本）。
   - 下发生成参数（目标年龄、页数范围、语言风格）。

3. **任务编排层（Celery/RQ/Temporal）**
   - 长任务异步执行：下载、切镜头、抽帧、ASR、摘要、排版。
   - 支持重试、断点续跑、状态回调。

4. **多模态处理服务**
   - 视频预处理服务（FFmpeg）。
   - 镜头检测服务（PySceneDetect/TransNetV2）。
   - 语音转写服务（Whisper 或云 ASR）。
   - 图像理解服务（Caption/OCR/人脸与角色线索）。
   - LLM 编排服务（剧情骨架、幼儿改写、转场优化）。

5. **存储层**
   - 对象存储（S3/MinIO）：原视频、关键帧、音轨、导出 PDF。
   - 关系库（PostgreSQL）：项目、场景、页面、任务、配额。
   - 可选向量库（pgvector）：场景语义检索、去重与召回。
   - Redis：任务队列、进度缓存、短期会话。

---

## 3. 端到端数据流（Pipeline）

### Step A：输入与标准化
1. 用户上传本地视频或提交 YouTube URL。
2. 统一转码为标准工作格式（建议 H.264 + AAC, 720p/1080p）。
3. 抽取音轨与基础媒体信息（时长、分辨率、帧率、码率）。

### Step B：场景切分与关键帧
1. 镜头切分（shot boundary detection）。
2. 相邻镜头聚合为语义场景（scene clustering）。
3. 每个场景选 1-3 张候选关键帧（清晰度、人物显著性、非黑屏）。

### Step C：文本证据构建
1. 读取字幕（若存在）或执行 ASR。
2. 将字幕按时间戳对齐到场景。
3. 对关键帧做图像描述（人物/动作/地点）。
4. 融合为“场景证据包”。

### Step D：叙事生成
1. 先生成“剧情骨架”（必保留关键节点）。
2. 将骨架映射到场景，筛选要出现在绘本中的页面。
3. 生成幼儿版短句（每页 1-2 句）。
4. 插入转场句（后来、第二天、就在这时）。
5. 执行一致性检查（人名、地点、时间、代词指代）。

### Step E：排版与导出
1. 生成页面布局（图 + 文 + 页码）。
2. 前端人工微调（换图/改句/重排）。
3. 导出 PDF、PNG 序列、可复编辑 JSON。

## 4. 数据模型设计（核心表）

建议在 PostgreSQL 中至少有以下实体：

- `projects`
  - `id`, `user_id`, `title`, `language`, `target_age`, `status`, `created_at`
- `assets`
  - `id`, `project_id`, `type`(video/audio/frame/pdf), `uri`, `duration_s`, `meta_json`
- `scenes`
  - `id`, `project_id`, `start_ms`, `end_ms`, `score`, `summary_raw`, `evidence_json`
- `scene_frames`
  - `id`, `scene_id`, `uri`, `timestamp_ms`, `quality_score`, `selected`
- `transcripts`
  - `id`, `project_id`, `start_ms`, `end_ms`, `text`, `speaker`
- `story_nodes`
  - `id`, `project_id`, `order_no`, `node_type`(setup/conflict/transition/...)
- `pages`
  - `id`, `project_id`, `page_no`, `scene_id`, `frame_id`, `text`, `transition_text`, `version`
- `jobs`
  - `id`, `project_id`, `job_type`, `status`, `progress`, `error_log`, `cost_json`

说明：
- 所有大文件只存对象存储 URI，不直接落库二进制。
- `evidence_json` 建议保存“字幕片段 + 关键帧描述 + 置信分”，便于回溯。

## 5. 视频规格与处理限制建议

### 5.1 输入建议
- 容器：`mp4`, `mkv`, `mov`
- 编码：H.264/H.265（若不兼容，先转码）
- 分辨率：建议 ≤ 1080p（4K 自动降采样）
- 帧率：建议 ≤ 30fps（高帧率素材先降帧）

### 5.2 v1 限制（建议）
- 单文件大小：≤ 4GB
- 单视频时长：≤ 150 分钟
- 单项目最大页数：建议 20-80 页（可配置）

### 5.3 成本与稳定性折中
- 长视频先“粗切分 + 低分辨率抽帧”，二次精炼只处理候选片段。
- 静态镜头可降低抽帧密度，动作场景提高抽帧密度。

## 6. Token 预算与成本控制（LLM）

### 6.1 Token 消耗来源
1. 场景证据（字幕 + 图像描述）输入。
2. 剧情骨架生成。
3. 页面文案生成/改写。
4. 一致性修复与重写。

### 6.2 估算方法（示例）
- 假设 120 分钟电影切成 250 个场景。
- 每场景输入摘要压缩到 120 tokens，则输入约 30,000 tokens。
- 剧情骨架 + 页面文案 + 修复多轮共 20,000-60,000 tokens。
- 总量粗估：50,000-100,000 tokens/部电影（取决于压缩策略）。

### 6.3 降本策略
- 两阶段摘要：先小模型压缩，再大模型做最终叙事。
- 场景去冗余：相似场景合并，减少重复 token。
- Prompt 模板固定化，限制输出长度与风格漂移。
- 人工编辑优先替代多轮自动重写。

## 7. 性能与时延预算（离线任务）

以 90-120 分钟视频为例，目标总耗时 8-25 分钟（取决于硬件与模型）：

- 下载/转码：2-8 分钟
- 切镜头+抽帧：2-6 分钟
- ASR：3-12 分钟
- 场景摘要与文案生成：1-6 分钟
- 导出：<1 分钟

优化优先级：
1. 转码与抽帧并行化。
2. ASR 与图像描述并发。
3. LLM 调用批处理（batch）与缓存。

---

## 8. 质量保障：如何确保“剧情完整+转场自然”

建立可量化 QA 指标：

1. **覆盖率**：核心剧情节点命中率（人工标注对比）
2. **连贯性**：相邻页实体一致率（角色/地点/时间）
3. **可读性**：平均句长、词汇难度、低龄可理解评分
4. **图文匹配度**：文本与关键帧语义一致度
5. **人工修改率**：每本书需要人工改动的页占比

上线前建议做 20-50 个样本回归评测。

## 9. 安全、版权与合规

- YouTube 或电影素材需由用户保证版权与授权。
- 默认私有项目，素材与导出仅本人可见。
- 可配置自动清理策略（如 7/30/90 天清理原视频）。
- 提供“删除项目即彻底删除对象存储文件”的流程。

## 10. 监控与运维

建议最小监控项：
- 任务成功率、平均耗时、失败类型分布。
- 单项目 token 用量与费用。
- GPU/CPU 使用率、队列堆积长度。
- 对象存储容量增长曲线。

日志建议：
- 结构化日志（project_id/job_id/scene_id 维度可追踪）。
- 错误日志保留原始模型响应片段（脱敏后）。

## 11. 推荐 v1 技术选型（务实版）

- 前端：Next.js + Tailwind
- 后端：FastAPI
- 队列：Celery + Redis
- 数据库：PostgreSQL

## 12. 里程碑建议

1. **M1（2 周）**：本地视频 -> 自动切分 -> 关键帧 + 文案 -> PDF。
2. **M2（2 周）**：剧情骨架/转场强化 + 编辑器可改图改字。
3. **M3（1-2 周）**：YouTube 输入 + 成本面板（token 与时长统计）。
4. **M4（持续）**：质量评测集与自动回归。

## 13. 关键风险与缓解

- **风险：剧情断裂** -> 采用“骨架先行 + 节点强制覆盖”。
- **风险：成本过高** -> 分级模型与摘要压缩。
- **风险：处理时间过长** -> 并行处理与缓存中间结果。
- **风险：儿童语言不合适** -> 固定年龄模板 + 安全词典过滤。

这套架构的核心是“先证据、后叙事、可编辑回路”，保证既自动化又可控。

## 14. PostgreSQL 成本与 Supabase 取舍

### 14.1 PostgreSQL 是否“会花钱”
- PostgreSQL 本身是开源软件，不收授权费。
- 成本通常来自部署与运行：云主机、磁盘、备份、监控、运维人力。
- 如果你本地单机部署（Docker Compose），数据库软件层面可接近零成本。

### 14.2 Supabase 会不会更好
Supabase 本质是“托管 PostgreSQL + 周边能力（Auth/Storage/Realtime）”，适合想快速上线的人：

- **优点**
  - 减少运维工作量，开箱即用。
  - 自带鉴权和对象存储，能加快 MVP。
  - 对前端开发友好，原型迭代快。

- **代价/限制**
  - 项目增长后，托管资源费用可上升。
  - 对计算密集型视频任务帮助有限（视频处理仍需单独 worker/GPU）。
  - 对“完全离线、本地优先”路线不是最优。

### 14.3 建议
- **你当前诉求（版权敏感 + 本地导出）**：优先“本地自部署 PostgreSQL + MinIO”。
- **若后续要做轻量云协作**：可把元数据迁移到 Supabase，视频处理仍放自有 worker。

## 15. 版权约束下的产品定位建议

### 15.1 定位主张：Local-first
针对电影/YouTube 素材存在版权不确定性，建议将产品定位为：

- 用户自己提供素材。
- 处理尽量在本地设备或用户自托管服务器进行。
- 默认只导出到用户本地（PDF/图片包/项目文件）。
- 平台不长期留存源视频。

这个定位可以显著降低法律与平台合规压力。

### 15.2 设备导出策略（电脑 + 手机）
- **电脑端**：直接导出 PDF、ZIP（图片+文案）、JSON 工程文件。
- **手机端**：
  - PWA/网页端导出到系统文件（iOS Files / Android Downloads）。
  - 增加“低分辨率预览包”和“高分辨率打印包”两档导出。
- **跨设备传递**：使用一次性局域网下载链接（默认 24 小时过期）。

## 16. 开源 Easy Setup vs 公有 SaaS 网站

### 16.1 两种模式对比

1. **开源 + Easy Setup（推荐先做）**
   - 优点：
     - 版权风险更可控（用户本地处理）。
     - Token 与算力成本由用户承担，不会平台爆炸。
     - 社区可贡献模型适配与优化。
   - 缺点：
     - 用户安装门槛需要降低（脚本/一键包）。
     - 商业化路径需另行设计（高级功能、托管版等）。

2. **公有 SaaS（输入链接即用）**
   - 优点：
     - 用户体验最好，增长快。
   - 缺点：
     - 版权与投诉风险高。
     - 视频处理 + LLM token + 存储三重成本，极易失控。
     - 需要风控、配额、审核与法务支撑。

### 16.2 现实路线（建议）
- **Phase 1**：开源本地版（桌面/自托管）
- **Phase 2**：半托管版（只托管项目元数据，不托管源视频）
- **Phase 3**：受限 SaaS（仅允许用户上传授权素材，并严格配额）

## 17. Token 与成本“爆炸”防护策略

若未来做在线版，建议一开始就实施硬性限制：

1. **配额系统**
   - 按用户/套餐限制月视频分钟数、页数、token 上限。
2. **分层模型路由**
   - 粗处理走低成本模型，最终润色才用高质量模型。
3. **缓存与去重**
   - 相同素材片段复用中间结果（ASR、场景摘要、关键帧描述）。
4. **可控输出模板**
   - 限制每页字数与风格，防止无意义长输出。
5. **超预算熔断**
   - 到达预算阈值自动暂停，要求用户确认继续。

## 18. 推荐你当前阶段的决策

结合你“给女儿做绘本”的真实需求，建议优先级如下：

1. 做 **开源 + 本地优先** 的 MVP。
2. 首先支持 **电脑导出 + 手机可下载查看**。
3. 默认不保存源视频到公共云，降低版权风险。
4. 成本层面先让用户使用本地模型或自备 API Key。
5. 等流程稳定后，再考虑受限 SaaS 化。

这样能最快获得可用成果，同时避免早期被版权与成本问题卡死。

## 19. 极简 MVP 路线（开源工具、Local-only、最小存储）

你的判断是对的：如果当前目标是“尽快发布可用 MVP 并 build in public”，可以先不做复杂持久化系统，采用**会话级处理 + 本地导出**。

### 19.1 为什么这样更容易出 MVP
- 少做数据库与账号体系，开发面显著收敛。
- 版权风险更低：不托管用户源视频，不做长期云存储。
- 关注真正价值：视频 -> 连贯故事绘本 的生成质量。

### 19.2 最小化存储策略（建议）
只保留运行必须的数据：
- 临时目录：中间帧、ASR文本、场景切分结果（任务完成后可自动清理）。
- 最终产物：PDF/图片包/项目JSON（用户手动保存）。
- 可选缓存：仅本机缓存最近 N 次处理结果（例如 7 天自动删除）。

**不做**（MVP阶段）：
- 多用户项目库
- 云端历史工程管理
- 长期元数据统计面板

### 19.3 但“故事高级连贯”必须保留（这是 MVP 核心）
即使简化存储，也要保留高质量叙事链路：

1. **剧情骨架先行**
   - 先从整片提取 10-20 个关键剧情节点，再映射到页面。
2. **强制转场生成**
   - 在节点间插入时间/地点/因果转场句，避免跳切叙事。
3. **一致性检查**
   - 检查角色称呼、地点变化、时序关系、代词指代。
4. **页面预算控制**
   - 每页固定短句长度，保证低龄可读和节奏稳定。
5. **人工可编辑回路**
   - 提供“换图/改句/重排”三件套，确保最后质量可控。

### 19.4 推荐的“先快后全”两阶段
- **阶段A（现在）**：开源本地工具版
  - 输入：本地视频
  - 输出：PDF + ZIP
  - 存储：仅临时文件 + 本地导出
- **阶段B（有需求再做）**：逐步增加项目管理/云协作
  - 仅在用户明确提出时再加数据库与托管能力

这个路线通常能更快拿到第一批真实用户反馈，同时不牺牲你最看重的“情节完整、转场自然”。
